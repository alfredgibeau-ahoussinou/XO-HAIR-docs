Figure 1 — Functional architecture of the XOhair Melun website

Reading the diagram: the client browses the storefront or goes directly to booking. Available slots are calculated in real time via the API (no CDN cache). Each request is stored in Netlify Blobs, triggers Brevo emails, and appears in Ola's admin. The deposit is paid via configurable PayPal or Revolut links.

  • No dependency on third-party booking SaaS (Calendly, SimplyBook, etc.)
  • Data hosted on Netlify Blobs — full control by Ola via the admin
  • Dual email channel (Brevo + fallback) for notification reliability

    participant B as Brevo

    S->>API: Checks availability
    API-->>S: Available slots
    C->>S: Submits form
    S->>API: Creates booking request
    API->>B: Request received email
    B-->>C: Confirmation
    API->>B: Ola alert
    B-->>O: Notification
    O->>S: Accepts or declines
    alt Accepted
        API->>B: Confirmation + deposit

    else Declined
        API->>B: Slot unavailable
        B-->>C: New slot suggested
    end

Figure 2 — Client journey from booking to confirmation

The client books independently, receives an immediate email, Ola validates from her admin, the client pays the deposit if accepted. The entire flow is automated — Ola only needs to accept or decline.
